## 1. 什么是 DPO——一个好比喻

### 1.1 从"老师批改作业"说起

想象一个场景：

学生已经学完课本（预训练），也做过练习题（SFT 监督微调），写得像模像样。但老师发现了一个问题——学生**不知道什么答案更好**。

于是老师拿出一沓卡片，每张卡片上有**两个答案**：

- 上面写的是 **好答案**（chosen），被圈了红勾
- 下面写的是 **差答案**（rejected），被打了红叉

老师对学生说："以后你要多往好答案的方向写，少往差答案的方向写。"

**这就是 DPO。** 它不告诉模型"标准答案是什么"，只告诉它"哪个更好，哪个更差"。

### 1.2 为什么需要 DPO？

SFT（监督微调）教会模型"怎么开口说话"，但它**没有偏好感知**：

```
用户问："怎么学习编程？"

SFT 模型可能回答：
A. "多写代码，多读代码，多请教有经验的人。"      ← 好的
B. "别学了，打游戏多好玩。"                      ← 差的

SFT 模型觉得 A 和 B "概率差不多"，因为它只学过"像人类的文字"，
没学过"哪个答案更有帮助"。
```

DPO 做的就是第二步校正：用对比数据让模型**拉开好答案和差答案的概率差距**。

### 1.3 DPO 在训练管线中的位置

```
预训练（学语言规律）
    ↓
SFT 监督微调（学会对话格式）
    ↓
DPO 偏好优化（学会区分好坏）
    ↓
最终模型（既能说人话，又知道什么更好）
```

> 注意：DPO 是 SFT 之后的步骤，模型必须先"会说话"才能"辨好坏"。跳过 SFT 直接做 DPO 效果会很差。

## 2. DPO 的数据长什么样

DPO 的训练数据不是单条文本，而是**成对的偏好数据**。每条数据包含：

| 字段 | 含义 | 示例 |
|------|------|------|
| `prompt` | 用户提问 | "解释一下量子纠缠" |
| `chosen` | 好答案 | "量子纠缠是量子力学现象……" |
| `rejected` | 差答案 | "不知道，你问别人吧" |

在 MiniMind 的代码中，数据以 JSONL 格式存储，`DPODataset` 会把它处理成 6 个张量：

```
x_chosen, y_chosen, mask_chosen       ← 好答案的输入、输出、mask
x_rejected, y_rejected, mask_rejected ← 差答案的输入、输出、mask
```

- `x` = prompt + answer 的拼接序列（token id）
- `y` = 预测目标（答案部分）
- `mask` = 只标记"答案部分"的 token，prompt 部分不计入 loss

**为什么需要 mask？** 因为 prompt 是用户给的，不需要模型"学会"什么才是好 prompt。模型只需要学会"什么是好答案"。

## 3. 直觉理解 DPO 公式

### 3.1 核心思想（先别看数学）

DPO 想让策略模型做到一件事：

> **让好答案的概率变大，让差答案的概率变小。**

但有个关键约束：**不能瞎改。** 如果模型完全放飞自我，答案虽然变好了，但语法和风格可能面目全非。所以需要一个"参考模型"（reference model）来兜底——它是冻结的 SFT 模型，代表了"原始风格"。

### 3.2 公式拆解

DPO 的损失函数：

```
L_DPO = -log(σ(β × (logπ(好答案|x) - logπ(坏答案|x) -
                    (logπ_ref(好答案|x) - logπ_ref(坏答案|x)))))
```

别怕，我们拆开看。令：

```
R(好) = logπ(好|x) - logπ_ref(好|x)    ← 策略和参考对"好答案"的差距
R(坏) = logπ(坏|x) - logπ_ref(坏|x)    ← 策略和参考对"坏答案"的差距
```

那么：

```
logits = R(好) - R(坏)
loss   = -log(sigmoid(β × logits))
```

### 3.3 直觉翻译

| 情况 | logits | sigmoid | loss | 说明 |
|------|--------|---------|------|------|
| 好答案概率 >> 差答案 | 正大 | 接近 1 | 接近 0 | 训练成功，模型"答对了" |
| 两者差不多 | 接近 0 | 等于 0.5 | 较大 | 模型还没学会区分 |
| 好答案 << 差答案 | 负大 | 接近 0 | 极大 | 模型学反了，要大力纠正 |

**训练目标就是让 loss 越来越小。** 当模型越来越倾向于给好答案打高分、给差答案打低分时，loss 自然下降。

## 4. 再看代码：loss 一步步拆解

### 4.1 第一步：把 logits 转成 token 级别的 log 概率

```python
def logits_to_log_probs(logits, labels):
    log_probs = F.log_softmax(logits, dim=2)              # [B, T, V] -> [B, T, V]
    log_probs_per_token = torch.gather(log_probs, dim=2,  # [B, T]
                                        index=labels.unsqueeze(2)).squeeze(-1)
    return log_probs_per_token                            # [B, T]
```

这一步做的事：模型输出的是每个位置的整个词表概率分布（logits）。但我只关心"正确答案对应的那个 token"的概率是多少，所以用 `gather` 把它拎出来。

### 4.2 第二步：用 mask 求和，得到整段答案的总 log 概率

```python
ref_log_probs = (ref_log_probs * mask).sum(dim=1)      # [2B] -> 每个样本一个标量
policy_log_probs = (policy_log_probs * mask).sum(dim=1)
```

mask 的作用：只保留答案部分的 token，prompt 部分乘以 0 不计入。然后沿着序列维度求和，得到每段答案的"总打分"。

> 为什么求和而不是平均？因为长答案如果每个 token 都略好一点点，加起来差距就更明显。

### 4.3 第三步：拆分 chosen / rejected

```python
batch_size = ref_log_probs.shape[0]
chosen_ref    = ref_log_probs[:batch_size // 2]    # 前半段是好答案
reject_ref    = ref_log_probs[batch_size // 2:]    # 后半段是差答案
chosen_policy = policy_log_probs[:batch_size // 2]
reject_policy = policy_log_probs[batch_size // 2:]
```

注意这里的设计技巧：chosen 和 rejected 被 **拼接在同一个 batch** 里传入模型，只分了一次就拿到了四个值。这样既省显存又高效。

### 4.4 第四步：算差值，算 loss

```python
pi_logratios  = chosen_policy - reject_policy      # 策略的偏好差距
ref_logratios = chosen_ref    - reject_ref         # 参考模型的偏好差距
logits        = pi_logratios - ref_logratios       # 策略比参考"多"好多少
loss          = -F.logsigmoid(beta * logits)       # DPO 损失
return loss.mean()
```

`logits` 的含义：策略模型比参考模型**更好地**区分了好坏答案。当这个值很大时，`sigmoid` 接近 1，`log(1) = 0`，loss 趋近于零——训练完成。

### 4.5 训练循环中的整体调用

```python
# 1. 参考模型前向（冻结，不计算梯度）
with torch.no_grad():
    ref_outputs = ref_model(x)
ref_log_probs = logits_to_log_probs(ref_outputs.logits, y)

# 2. 策略模型前向（正常计算梯度）
outputs = model(x)
policy_log_probs = logits_to_log_probs(outputs.logits, y)

# 3. 算 DPO 损失
dpo_loss_val = dpo_loss(ref_log_probs, policy_log_probs, mask, beta=args.beta)

# 4. 反向传播
loss = (dpo_loss_val + outputs.aux_loss) / args.accumulation_steps
scaler.scale(loss).backward()
```

关键点：
- 参考模型在 `no_grad()` 下运行，不占用梯度内存
- MoE 结构还有额外的 `aux_loss`（负载均衡损失），一起加进去
- 支持梯度累积（`accumulation_steps`），batch 小的时候可以攒几步一起更新

## 6. 超参数重点：为什么学习率这么低？

三个训练阶段的学习率对比：

```
预训练： 5e-4  ──┐
                  ├─ 差距 1 万倍以上
SFT：     1e-5  ──┤
DPO：     4e-8  ──┘
```

为什么跨度这么大？

### 6.1 原因一：灾难性遗忘

DPO 之前模型已经学得很好了（SFT 教会了对话格式和指令遵循）。如果学习率太大，模型会**忘记之前的知识**。这就像老师批改作业时用橡皮猛擦——擦太用力把原来写对的内容也擦掉了。

### 6.2 原因二：数据量小

DPO 偏好数据通常远少于 SFT 数据。预训练可能有数十亿 token，SFT 有数十万条对话，而 DPO 偏好对可能只有几千到几万条。数据少 + 学习率大 = 过拟合风险极高。

### 6.3 原因三：只需微调

DPO 的目标不是"重写"模型，只是"微调偏好方向"。好答案和差答案的概率在 SFT 后已经有一定差距，DPO 只是把差距拉大。

> **经验法则**：DPO 的学习率大约是 SFT 的 1/100 ~ 1/1000。MiniMind 用的是 4e-8。

### 6.4 其他关键超参数

| 参数 | 默认值 | 说明 |
|------|-------:|------|
| `--epochs` | 1 | 偏好数据看一遍就够了，多刷容易过拟合 |
| `--batch_size` | 4 | 每条数据含 chosen+rejected 两份，实际占显存较多 |
| `--max_seq_len` | 1024 | 偏好回复通常较长，需要足够窗口 |
| `--beta` | 0.15 | 惩罚强度（见下一节） |
| `--grad_clip` | 1.0 | 梯度裁剪，防止学习率极低但梯度爆炸 |
| `--save_interval` | 100 | 频繁保存，因为 DPO 不稳定时回退代价小 |

## 7. beta 参数：惩罚的"力度旋钮"

### 7.1 beta 到底是什么

公式中的 β 控制了**参考模型的约束有多强**。它可以被理解为 KL 惩罚的系数：

```
loss = -log(sigmoid(β × logits))
         ↑
    β 越大，sigmoid 越陡峭
```

### 7.2 不同 β 的效果

| β 值 | 行为 | 类比 |
|------|------|------|
| 0.01 | 几乎不管参考模型，策略自由发挥 | 老师放手让学生自己写 |
| **0.15**（默认） | 适度约束，平衡质量和风格 | 老师适当提醒"别跑题" |
| 0.5 | 严格贴合参考模型，变化很小 | 老师盯着每个字改 |
| 1.0+ | 几乎不让模型偏离 SFT 状态 | 老师橡皮不离手 |

### 7.3 MiniMind 为什么选 0.15

- 模型很小（64M 参数），对超参数不敏感，0.15 是经验值
- 推荐范围 0.1 ~ 0.5，根据数据和任务调整
- 如果模型质量下降（回答变短/退步），尝试调小 β
- 如果模型改变不够明显（好坏答案区别不大），尝试调大 β

## 8. DPO vs PPO 一句话总结

| 维度 | DPO | PPO |
|------|-----|-----|
| 思路 | 直接用偏好数据当"正确答案" | 先用偏好数据训练奖励模型，再用 RL 优化 |
| 复杂度 | 低（一个 loss 搞定） | 高（奖励模型 + 价值函数 + GAE） |
| 显存 | 只需多一个冻结参考模型 | 需要多个模型 + buffer |
| 稳定性 | 高，确定性优化 | 低，RL 训练波动大 |
| 效果 | 通常与 PPO 相当甚至更好 | 经典但复杂 |

一句话：**DPO 用更简单的方式达到了与 PPO 相当的效果。** 这也是为什么 MiniMind 默认使用 DPO 做偏好对齐。
